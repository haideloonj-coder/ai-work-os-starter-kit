# Contributing

Thank you for helping improve this Markdown-first public starter kit. Contributions should make the repository easier to understand, safer to adapt, or more useful for repeatable AI-assisted work.

## Good Contribution Types

- README and onboarding improvements
- Clearer templates and handoff prompts
- Fictional sample workflows
- Review checklists and safety checks
- Documentation fixes, examples, and small clarifications

## Keep Contributions Public-Safe

Do not add credentials, API keys, OAuth tokens, company data, contracts, customer records, private metrics, internal logs, personal data, local file paths, private storage links, or real unreleased business plans.

Use fictional examples or public-safe placeholders only. When in doubt, remove the detail or label it as fictional.

## Before Opening a Pull Request

- Read `README.md` and `00_system/update_rules.md`.
- Keep the change small and focused.
- Preserve the existing folder structure unless the change clearly requires otherwise.
- Add or update maintenance notes when the change is meaningful.
- Check that all examples are fictional or public-safe.

## Suggested Review Checklist

- [ ] No credentials, secrets, tokens, or private links were added.
- [ ] No company data, customer records, private metrics, or internal logs were added.
- [ ] Examples are fictional or clearly public-safe.
- [ ] Links point to repository-local files or public resources.
- [ ] Language is concise, practical, and non-hype.
- [ ] The change is easy for non-engineers to understand.

## Style Guide

- Prefer plain Markdown.
- Use short sections and direct headings.
- Keep examples copy-paste friendly.
- Mark uncertainty instead of inventing facts.
- Separate sources, indexes, templates, routines, and examples.
- Avoid large rewrites when a small edit is enough.
